# 1.1 데이터 / 데이터베이스 / 데이터베이스 시스템
- 데이터: 사실을 나타내는 수치이며 관찰의 결과로 정량적, 정성적으로 표시가 가능한 실존하는 값
- 정보: 데이터 중에 의미 있는 데이터를 정보라 한다
- 지식: 사물이나 현상에 대해 이해한 것
## 데이터베이스 / 데이터베이스 시스템
- DATASTORE: 데이터가 저장된 매체 ex) 엑셀, 엑세스, 문서, 메일, DBMS ,..
- DATABASE: 데이터를 체계적으로 수집, 관리, 통합해둔 것 ex) 엑셀, 엑세스, 문서, 메일, DBMS ,..
- DATABASE SYSTEM: 체계적으로 데이터를 수집하고 통합, 공유, 활용할 수 있게 해주는 S/W, H/W 시스템
## 데이터베이스 시스템 구성요소
- DBMS: 사용자와 데이터베이스를 연결시켜주는 S/W
- DATABASE: 데이터가 저장되는 장소나 장치
- DATA Model: 데이터를 저장하는 이론적이고 논리적인 기술과 방법
- RDBMS: 관계형 데이터 베이스

# 1.2 데이터 모델
## 데이터모델
- 데이터가 데이터베이스에 저장되는 방법
- **데이터베이스에 데이터가 어떻게 구조화(정형)되어 저장되는지 결정하는 방법**
## 데이터모델 유형
- 계층데이터모델, 네트워크데이터모델, 객체데이터모델, 관계데이터모델 등
- 데이터 간의 관계를 어떤 방식으로 표현하는가에 따라 차이
## 관계 데이터 모델 (Relational Data Model)
- 데이터내에 존재하는 속성값을 기준으로 데이터 간의 관계를 설정
- ex) 고객정보(고객아이디, 고객명)

# 1.3 관계 데이터 모델
- 테이블 형태로 자료들간의 관계를 표현하는 모델링(설계) 방식
- 집합 형태의 테이블 단위로 데이터를 처리하여 데이터간의 구조를 이해하고 접근하기 용이하게 함
### 주요 개념
- Relation: 행과 열로 구성된 **테이블**
- Table: Relation의 시각적 표현 방법
- 데이터와 테이블(Relation): 테이블이 Relation이고 테이블 내 원소들의 값이 데이터
- Relationship: 테이블내 또는 테이블 간의 관계
- Relationship 유형: 테이블내의 관계(컬럼값의 집합), 테이블간의 관계(관계 키값을 통해 연결)
### Relation(Table) 주요 구성 요소
- Relation(테이블=Entity) 구성요소: 스키마와 인스턴스
- Scheme: 관계형 데이터베이스내 릴레이션 구조를 정의한 것(전체 로우)
- Instance: 테이블 스키마에 저장된 실제 데이터 셋
- Attribute(속성): Column, Field라고도 함, 속성 값과 값의 범위를 도메인이라하고 속성의 수를 차수라 한다
- Row: Relation 인스턴스의 각행, Tuple, Record라고도 말하고 행의 수를 Cardinality라고 함. 각 행의 값은 고유해야 하며 중복된 행이 존재해서는 안된다
### 관계 데이터모델 표현 방식
- 테이블: 릴레이션을 통해 테이블로 표현
- 제약조건: 테이블에 대해 제약조건을 통해 데이터 무결성을 관리
- 관계대수: 관계대수(테이블 간의 관계정의)를 통해 원하는 구조의 값을 구할 수 있다 - join
### 관계 데이터 모델 설계 방법
- UML(Unified Modeling Language) 기반 ERD(Entity Relationship Diagram) 등을 통해 테이블, 제약조선, 관계대수 등을 분석/설계
- ERD 툴: 데이터 모델링 전문 툴: ERWIN, UML 툴: StarUML, Visio, RDBMS 모델링 기능
### 관계 데이터 모델 개발 방법
- RDBMS: 관계형 데이터 베이스 시스템(Oracle, Ms SQL, MySQL..) 기반에서 관계 데이터 모델 개발
- SQL 쿼리 언어를 이용해 테이블, 제약조건, 관계대수를 설정하고 데이터를 관리할 수 있다
# 1.4 제약조건
## 키 key
- 다른 값과 구분이 가능한 유일한 값
- 키는 속성(컬럼)들에 할당됨
- 로우들 간의 구분 값(PK, UK)으로 사용되거나 테이블간 관계설정시 참조값(FK)으로 사용됨

- 슈퍼키: 행을 서로간 구분할 수 있는 속성 또는 속성들의 집합 
- 후보키: 행을 구분할 수 있는 속성의 최소 집합
- 복합키: 두 개의 속성으로 구성된 행을 구분할 수 있는 키조합 ex) 주문상품정보: 주문번호+주문상품번호
- 유니크키: 단일 컬럼 내 값이 중복되지 않은 컬럼
- 기본키: PrimaryKey 후보키 중 대표로 삼는 키, 슈퍼키이자 후보키이자 유니크키 또는 복합키, 저장 후 변경 불가한 키, 노출해도 무방한 키
- 대리키: 기본키의 일종으로 일련번호 같은 고유키 항목 설정이 어려운 경우 자동 채번 기능으로 생성
- 외래키: Foreign Key, FK 특정 테이블의 기본키를 참조하는 참조키, 참조키를 이용해 테이블간의 관계를 설정

### 데이터 무결성 원칙
- DB에 저장된 데이터는 결함이 없어야 한다는 원칙
- 데이터의 일관성과 정확성을 제공해야한다
### 무결성 제약조건
- 데이터 무결성을 지키기 위한 수단 제공
- 대부분의 DBMS에서는 데이터 등록/수정/삭제 시 데이터 무결성을 담보할 수 있는 제약조건을 통해 데이터 무결성 관리
- 주로 컬럼의 데이터 입력 제약(데이터 형식, 기본값, 널값) 제공과 주요 키를 활용해 제약조건 수단을 제공
### 무결성 제약조건 유형
- 도메인 무결성 제약조건: 테이블 속성 값에는 도메인의 지정된 형식의 값들이 저장될 수 있게하여 컬럼들의 무결성을 유지(ex) 주문일시: 반드시 날짜일시 
- 개체(Entity=Table) 무결성 제약조건: 모든 테이블에는 기본키를 설정해 행간 데이터 중복을 방지
- 참조 무결성 제약조건: 외래키를 통해 테이블간 참조관계를 설정하여 테이블간 데이터 무결성을 유지
- 유니크 제약조건: 유니크키를 통해 해당 컬럼 내 동일값이 존재하지 않게 제약하여 무결성을 유지
# 1.5 관계유형(관계대수)
### 관계대수
- 단일 릴레이션(테이블) 또는 복수 테이블 관계에서 데이터를 추출하는 방법 제공
- 데이터 추출을 위한 구조화된 질의 언어(SQL)를 통해 처리
### 관계대수 주요 연산 유형
- 선택연산(Selection): 하나의 테이블 로우 데이터 집합을 구하는 연산, 다양한 조건식 제공
	- Select * from Members Where age > 19
- 프로젝션(Projection): 하나의 테이블의 특정 컬럼 로우집합을 구하는 연산
	- Select ID, Name From Members Where age > 19
- 집합연산: 여러 테이블을 대상으로 하며 합집합, 교집합, 차집합, 카티션 프로덕트 유형이 있음
### 수직적 집합연산: Union 
-  (분류별로 나누어진 상품 정보)
- 집합연산 합집합: 모든 행을 합하여 하나의 테이블 형태로 제공, 중복되는 행은 하나만 표현
- 집합연산 교집합: 공통 행의 집합을 제공, 중복되는 행 데이터만 조회
- 집합연산 차집합
### 수평적 집합연산: Left, Right, Full join
- 카티션 프로덕션: 무조건 두개 테이블을 수평적으로 하나의 테이블로 합치는 방법
- 조인: 두 테이블간 공통속성을 기준으로 수평적으로 하나의 테이블로 합치는 방법
### 조인의 종류
- 세타조인: 속성값을 비교하여 비교 조건에 만족하는 행만 가져오는 조인(모든 조건 가능)
- 동등조인: 속성값을 비교하여 동일한 값에 해당하는 행만 가져오는 조인(동일 조건만 가능)
- 자연조인: 동등조인과 동일하지만 두 테이블의 조인 속성 중 하나만 출력됨. Inner Join
- 외부조인: 자연조인 하되 조인에 실패한 행의 값을 null로 표시. Left/Right/Full Outer Join
- 세미조인
- 디비전

# 2.1 데이터 모델링 절차
### 데이터 모델링
- 데이터베이스 분석 설계 과정
- 현실세계의 복잡한 개념을 단순화/추상화(일반화)시켜 데이터베이스화 하는 과정
- 모델링이란 사용자 요구사항을 받아 분석/설계하는 개발 전 단계까지의 모든 과정
### 데이터 모델링 주요 절차
- 개념적 모델링: 현실 세계에 존재하는 개념을 추상화한 **개체를 도출**하고 업무프로세스를 정의
	- 주요 개념 도출: 주요 개체(Entity) 도출, 개체의 PK 등 기본 속성 도출, 개체들간의 관계를 ERD로 작성
- 논리적 모델링: 정규화 과정을 통해 개념모델과 업무프로세스를 논리적(한글) 설계하여 테이블과 테이블간 관계를 정의
	- 주요 개념 구체화: 관계데이터모델 활용 구체화, 상세 속성 정의, 정규화
- 물리적 모델링: 논리적 모델링 산출물을 RDBMS 환경에 맞게 실제 물리적 테이블(영문)을 설계하고 테이블 간 관계를 정의
	- RDBMS 기반 실제 DB 객체 설계: 물리적 DB 개체 정의(Table, View, Index, Key..)
# 2.2 ER 모델 

### ER(Entity Relationship) 모델: 엔티티 관계모델
- 개념 모델링 단계에서 사용되며 도출된 개체간 관계정의에 사용되는 모델링 방식
- 주로 고객과의 쉬운 상호 이해를 돕기 위해 다이어그램 방식으로 표현
- 표준화된 그림으로 표현방식이 ERD
### ER 관계 유형
- 차수에 따른 유형
	- 1진 관계: 테이블 본인 한 개와의 관계
	- 2진 관계: 두 개의 테이블간 관계
	- 3진 관계: 세개 테이블간 관계
- 관계 대응수에 따른 유형: 개체들간 관계에서 개체 내 개별 개체(로우)들간의 대응관계 1:1, 1:N, N:1, M:N
### ER 관계 대응수에 따른 유형
- 1:1: 개체간 개별개체가 일대일로 존재 ex) 직원-사용관계-사용증
- 1:N, N:1: 단일개체가 다른 개체에 여러 개 존재하는 경우 ex) 부서-소속관계-직원
- M:N: 개별개체가 상호 여러 개 존재하는 경우 ex) 사용자-주문관계-상품
# 2.4 정규화
## 이상현상과 정규화 과정
- 이상현상 = 테이블 설계에 문제가 있어 데이터베이스 일관성과 데이터 무결성을 깨뜨려 발생되는 문제
- 이상현상을 없애기 위한 방법으로 정해진 규칙에 따라 테이블을 분해하는 과정을 정규화라고 함
- 정규화 과정이란 이상 현상을 발생시키는 테이블에 대해 이상현상이 없어질 때까지 해당 테이블을 분해하는 과정
### 정규화 과정
- 1정규화: 원자값 원칙, 단일 컬럼 단일값 제공 원칙, 기본키 존재, 일반 속성들은 기본키에 종속적이어야 함 ex) 회원정보에서 회원 취미 정보는 회원기본정보에서 분리
- 2정규화: 기본키가 2개 이상으로 구성된 테이블인 경우 일반 속성이 PK 속성들 중 일부 속성에 대해서만 종속적일 경우 해당 속성을 제거하고 별도 테이블로 분리 ex) 장바구니 상품정보를 장바구니와 상품정보로 분리
- 3정규화: 테이블 내 일반 속성들이 기본키 외에 다른 일반 속성에 종속적일 때 일반 속성에 종속되는 속성을 제거하여 테이블로 분리 ex) 주문정보와 배송료 정보
